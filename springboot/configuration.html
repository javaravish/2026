    Spring Boot Configuration and Properties Files

    1. Application Properties

    Spring Boot uses application.properties or application.yml files for configuration. By default, these files are located in the src/main/resources directory.

    server.port=8080
    spring.datasource.url=jdbc:mysql://localhost:3306/mydb
    spring.datasource.username=user
    spring.datasource.password=password

    2. Application YAML

    Alternatively, you can use application.yml for configuration:

    server:
        port: 8080
    spring:
        datasource:
            url: jdbc:mysql://localhost:3306/mydb
            username: user
            password: password

    3. Profiles

    Spring Boot allows you to define different configurations for different environments using profiles. For example, you can create application-dev.properties for development and application-prod.properties for production.

    spring.profiles.active=dev

    4. Accessing Properties in Code

    To access properties in your Spring Boot application, you can use the @Value annotation or the Environment abstraction.

    import org.springframework.beans.factory.annotation.Value;
    import org.springframework.stereotype.Component;

    @Component
    public class MyComponent {
        @Value("${server.port}")
        private int port;

        public void printPort() {
            System.out.println("Port: " + port);
        }
    }
    import org.springframework.core.env.Environment;
    import org.springframework.beans.factory.annotation.Autowired;
    import org.springframework.stereotype.Component;

    @Component
    public class MyComponent {
        @Autowired
        private Environment env;

        public void printPort() {
            String port = env.getProperty("server.port");
            System.out.println("Port: " + port);
        }
    }

    5. Using Configuration Classes

    You can also create configuration classes to group related properties together using the @ConfigurationProperties annotation.

    import org.springframework.boot.context.properties.ConfigurationProperties;
    import org.springframework.stereotype.Component;

    @Component
    @ConfigurationProperties("myapp")
    public class MyAppProperties {
        private String name;
        private int port;

        public String getName() {
            return name;
        }

        public void setName(String name) {
            this.name = name;
        }

        public int getPort() {
            return port;
        }

        public void setPort(int port) {
            this.port = port;
        }
    }

    6. Custom Property Sources

    You can also add custom property sources using the @PropertySource annotation. This is useful for loading properties from external files.

    import org.springframework.context.annotation.Configuration;
    import org.springframework.context.annotation.PropertySource;

    @Configuration
    @PropertySource("classpath:custom.properties")
    public class AppConfig {
    }

    Reading Values in a Spring Boot Application

    1. Using the `@Value` Annotation

    The @Value annotation is used to inject values from properties files directly into fields in Spring-managed beans.

    import org.springframework.beans.factory.annotation.Value;
    import org.springframework.stereotype.Component;

    @Component
    public class MyComponent {
        @Value("${server.port}")
        private int port;

        public void printPort() {
            System.out.println("Port: " + port);
        }
    }

    2. Using the `Environment` Abstraction

    The Environment abstraction provides a more flexible way to access properties and environment-specific configurations.

    import org.springframework.core.env.Environment;
    import org.springframework.beans.factory.annotation.Autowired;
    import org.springframework.stereotype.Component;

    @Component
    public class MyComponent {
        @Autowired
        private Environment env;

        public void printPort() {
            String port = env.getProperty("server.port");
            System.out.println("Port: " + port);
        }
    }

    3. Using `@ConfigurationProperties` Annotation

    The @ConfigurationProperties annotation allows you to bind properties from the configuration file to a POJO, grouping related properties together.

    import org.springframework.boot.context.properties.ConfigurationProperties;
    import org.springframework.stereotype.Component;

    @Component
    @ConfigurationProperties("myapp")
    public class MyAppProperties {
        private String name;
        private int port;

        public String getName() {
            return name;
        }

        public void setName(String name) {
            this.name = name;
        }

        public int getPort() {
            return port;
        }

        public void setPort(int port) {
            this.port = port;
        }
    }

    4. Using `@PropertySource` for Custom Property Files

    The @PropertySource annotation is used to specify additional property files to be loaded by Spring.

    import org.springframework.context.annotation.Configuration;
    import org.springframework.context.annotation.PropertySource;

    @Configuration
    @PropertySource("classpath:custom.properties")
    public class AppConfig {
    }

    Loading Environment-Specific Files in Spring Boot

    1. Profile-Specific Configuration Files

    Spring Boot supports profile-specific configuration files. You can define configuration files for different profiles by using a naming convention.

    # application.properties
    server.port=8080
    spring.datasource.url=jdbc:mysql://localhost:3306/mydb
    # application-dev.properties
    server.port=8081
    spring.datasource.url=jdbc:mysql://localhost:3306/devdb
    # application-prod.properties
    server.port=80
    spring.datasource.url=jdbc:mysql://localhost:3306/proddb

    2. Activating Profiles

    You can activate a specific profile in various ways:

    # In application.properties
    spring.profiles.active=dev
    # In application.yml
    spring:
        profiles:
            active: dev
    # As a command-line argument
    --spring.profiles.active=prod
    # As an environment variable
    SPRING_PROFILES_ACTIVE=prod

    3. Customizing Profile-Specific File Loading

    In addition to the default behavior, you can load custom property files based on the active profile using the @PropertySource annotation.

    import org.springframework.context.annotation.Configuration;
    import org.springframework.context.annotation.PropertySource;
    import org.springframework.core.env.Environment;

    @Configuration
    public class AppConfig {
        @PropertySource("classpath:application-${spring.profiles.active}.properties")
        public void loadProperties(Environment env) {
            String profile = env.getProperty("spring.profiles.active");
            System.out.println("Active profile: " + profile);
        }
    }

    @EnableConfigurationProperties in Spring Boot

    What is @EnableConfigurationProperties?

    The @EnableConfigurationProperties annotation is used in Spring Boot to enable support for @ConfigurationProperties. This annotation allows you to bind external configuration properties (from application.properties or application.yml files) to a Java bean.

    When to Use @EnableConfigurationProperties

    • Binding Configuration Properties to POJOs: Use @EnableConfigurationProperties when you have a configuration class annotated with @ConfigurationProperties that needs to be registered with the Spring context. This is particularly useful for binding related properties to a single configuration class.
    • Enabling Configuration Properties in Non-Spring Boot Applications: If you are using Spring Framework (without Spring Boot) or a custom Spring Boot setup where @ConfigurationProperties is not automatically enabled, you need to use @EnableConfigurationProperties to make it work.
    • Custom Configuration: When you need to create custom configuration classes for specific use cases and inject these configurations as beans, @EnableConfigurationProperties ensures that these properties are correctly bound and available in the application context.

    How to Use @EnableConfigurationProperties

    1. Create a Configuration Properties Class

    Define a POJO with fields corresponding to the configuration properties. Annotate this class with @ConfigurationProperties and @Component or @Configuration.

    @Component @ConfigurationProperties(prefix = "myapp")

    public class MyAppProperties {
    private String url;
    private int timeout;
    // Getters and Setters

    public String getUrl() {
    return url;
    }

    public void setUrl(String url) {
    this.url = url;
    }

    public int getTimeout() {
    return timeout;
    }

    public void setTimeout(int timeout) {
    this.timeout = timeout;
    }
    }

    2. Define the Properties in application.properties or application.yml

    # application.properties myapp.url=http://example.com myapp.timeout=5000
    # application.yml myapp: url: http://example.com timeout: 5000

    3. Enable Configuration Properties in Your Main Application Class

    If you are using Spring Boot, you usually don't need to explicitly use @EnableConfigurationProperties as Spring Boot auto-configures it for you. However, if you are configuring it manually or in a non-Spring Boot application, use @EnableConfigurationProperties in your configuration class.

    import org.springframework.boot.SpringApplication;
    import org.springframework.boot.autoconfigure.SpringBootApplication;
    import org.springframework.boot.context.properties.EnableConfigurationProperties;

    @SpringBootApplication
    @EnableConfigurationProperties(MyAppProperties.class)

    public class MyApplication {
    public static void main(String[] args) {
    SpringApplication.run(MyApplication.class, args);
    }
    }

    4. Inject the Configuration Properties Bean

    You can now inject the MyAppProperties bean into your Spring components.

    import org.springframework.beans.factory.annotation.Autowired;
    import org.springframework.stereotype.Service;

    @Service
    public class MyService {
    private final MyAppProperties properties;

    @Autowired
    public MyService(MyAppProperties properties) {
    this.properties = properties;
    }

    public void printProperties() {
    System.out.println("URL: " + properties.getUrl());
    System.out.println("Timeout: " + properties.getTimeout());
    }
    }

    Summary

    • What: @EnableConfigurationProperties is used to enable support for @ConfigurationProperties beans in a Spring application.
    • When to Use:
      • To bind external configuration properties to a Java bean.
      • When using Spring (without Spring Boot) or custom Spring Boot setups.
      • To inject configuration properties into Spring components.
    • How:
      • Create a configuration properties class annotated with @ConfigurationProperties.
      • Define the properties in application.properties or application.yml.
      • Use @EnableConfigurationProperties to enable property binding.
      • Inject the configuration properties bean where needed.

    This approach ensures that your application’s configuration is externalized, making it more flexible and easier to manage across different environments.

    Auto-configuration in Spring Boot is a mechanism that attempts to automatically configure your Spring application based on the dependencies present on the classpath. It is a powerful feature that simplifies the setup of Spring applications by providing sensible defaults.

    How Auto-Configuration Works Internally

    Spring Boot auto-configuration works through the following mechanisms:

    • Conditional Configuration: Auto-configuration classes use @Conditional annotations to conditionally apply configurations based on the presence of certain classes or properties.
    • Spring Factories: The META-INF/spring.factories file is used to register auto-configuration classes. Spring Boot reads this file to discover available auto-configurations.
    • Classpath Scanning: Spring Boot scans the classpath for certain classes and configurations to determine which auto-configuration classes should be applied.

    Enabling and Disabling Auto-Configuration

    Spring Boot provides flexibility to enable or disable auto-configuration as needed. Here’s how you can manage auto-configuration:

    1. Enable Auto-Configuration

    By default, Spring Boot auto-configuration is enabled. When you use the @SpringBootApplication annotation, it includes @EnableAutoConfiguration which activates auto-configuration.

    @SpringBootApplication
    public class MyApplication {

        public static void main(String[] args) {
            SpringApplication.run(MyApplication.class, args);
        }
    }

    2. Disable Auto-Configuration

    To disable specific auto-configuration classes, you can use the @SpringBootApplication annotation’s exclude attribute or specify exclusions in the application.properties or application.yml file.

    Using the `exclude` Attribute

    @SpringBootApplication(
        exclude = DataSourceAutoConfiguration.class
    )

    Using `application.properties` or `application.yml`

    To exclude auto-configuration classes in application.properties:

    spring.autoconfigure.exclude=
        org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration

    To exclude auto-configuration classes in application.yml:

    spring:
        autoconfigure:
            exclude:
                org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration

    3. Conditional Configuration

    Auto-configuration classes use various @Conditional annotations to determine if they should be applied. For example:

    @Configuration
    @ConditionalOnClass(DataSource.class)
    public class DataSourceConfiguration {

        @Bean
        public DataSource dataSource() {
            return new DataSource();
        }
    }

    The above code shows how a bean will only be created if the DataSource class is present on the classpath, using the @ConditionalOnClass annotation.

    Note

    Spring Boot automatically loads properties from application.properties or application.yml. Use @ConfigurationProperties for grouping properties and @PropertySource for custom property files.

    Spring Boot starters are a set of pre-configured dependencies and settings that simplify the setup of common functionalities in Spring Boot applications. They are designed to reduce the complexity of managing dependencies and configurations by bundling them into cohesive units.

    Why Use Spring Boot Starters?

    Starters help developers by:

    • Simplifying Dependency Management: Instead of manually adding multiple dependencies, you can include a starter that bundles all necessary libraries.
    • Providing Consistent Defaults: Starters come with sensible default configurations that can be customized as needed.
    • Encouraging Best Practices: By using starters, you follow established conventions and best practices that are already integrated into the starters.

    Common Starters

    Here are a few commonly used Spring Boot starters:

    • spring-boot-starter-web: Includes dependencies for building web applications. It includes Spring MVC, Jackson, and an embedded Tomcat server.
    • spring-boot-starter-data-jpa: Provides support for JPA (Java Persistence API) and Hibernate. It includes libraries for data access and transaction management.
    • spring-boot-starter-security: Adds security features to your application, including authentication and authorization, using Spring Security.
    • spring-boot-starter-test: Contains testing libraries like JUnit, Hamcrest, and Mockito, making it easier to write and run tests.

    Creating Custom Starters

    Creating a custom starter allows you to package a set of libraries and configurations into a reusable module. This is useful for sharing common setups across multiple projects.

    Steps to Create a Custom Starter

    1. Set Up Your Custom Starter Project

    Create a new Maven or Gradle project for your custom starter.

    mvn archetype:generate -DgroupId=com.example -DartifactId=my-custom-starter -DarchetypeArtifactId=maven-archetype-quickstart -DinteractiveMode=false

    2. Add Dependencies

    Add the dependencies your starter will use in the build configuration file:

    For Maven:
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter</artifactId>
        </dependency>
    </dependencies>
    For Gradle:
    dependencies {
        implementation 'org.springframework.boot:spring-boot-starter'
    }

    3. Create Auto-Configuration Classes

    Define configuration classes that will be applied automatically by Spring Boot:

    @Configuration
    public class MyCustomAutoConfiguration {

        @Bean
        public SomeBean someBean() {
            return new SomeBean();
        }
    }

    4. Create `META-INF/spring.factories`

    This file registers your auto-configuration classes with Spring Boot:

    org.springframework.boot.autoconfigure.EnableAutoConfiguration=
        com.example.MyCustomAutoConfiguration

    5. Package and Publish Your Starter

    Build and publish your starter so it can be used in other projects:

    mvn clean install

    Example Usage of Custom Starter

    To use your custom starter in a Spring Boot application, follow these steps:

    1. Add the Starter Dependency

    Include the custom starter in your application’s build configuration:

    <dependencies>
        <dependency>
            <groupId>com.example</groupId>
            <artifactId>my-custom-starter</artifactId>
            <version>1.0.0</version>
        </dependency>
    </dependencies>

    2. Use Auto-Configured Beans

    Utilize the beans configured by your starter in your application:

    @RestController
    public class MyController {

        private final SomeBean someBean;

        @Autowired
        public MyController(SomeBean someBean) {
            this.someBean = someBean;
        }

        @GetMapping("/bean")
        public String getBean() {
            return someBean.toString();
        }
    }
