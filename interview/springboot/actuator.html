    Spring Boot Actuator FAQ

    1. What is Spring Boot Actuator?

    Example Answer: Spring Boot Actuator is a project that provides production-ready features to help you monitor and manage your Spring Boot application. It exposes endpoints that give insights into the application's health, metrics, and other operational information, making it easier to manage and monitor applications in production.

    2. What are some common Actuator endpoints, and what do they do?

    Example Answer: Common Actuator endpoints include:
    • /actuator/health: Provides health status of the application and its dependencies (e.g., databases, external services).
    • /actuator/metrics: Exposes various metrics related to the application's performance, such as JVM metrics, application-specific metrics, and more.
    • /actuator/info: Displays arbitrary application information like build version, description, or custom metadata.
    • /actuator/env: Shows environment properties and configuration properties of the application.

    3. How do you enable Spring Boot Actuator in your application?

    Example Answer: To enable Actuator in a Spring Boot application, you need to add the Spring Boot Actuator dependency to your pom.xml (for Maven) or build.gradle (for Gradle):

    For Maven:

    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-actuator</artifactId>
    </dependency>

    For Gradle:

    implementation 'org.springframework.boot:spring-boot-starter-actuator'
    Actuator endpoints are then available by default. You can configure which endpoints are enabled or exposed in your application.properties or application.yml.

    4. How can you configure which Actuator endpoints are exposed?

    Example Answer: You can configure endpoint exposure in application.properties or application.yml:

    In application.properties:

    management.endpoints.web.exposure.include=health,info

    In application.yml:

    management:
      endpoints:
        web:
          exposure:
            include: health, info
    This configuration will expose only the /actuator/health and /actuator/info endpoints.

    5. How do you secure Actuator endpoints?

    Example Answer: Securing Actuator endpoints is typically done using Spring Security. You can restrict access to these endpoints by configuring security settings. For example:

    1. Add Spring Security dependency:

    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-security</artifactId>
    </dependency>

    2. Configure security settings:

    import org.springframework.context.annotation.Bean;
    import org.springframework.context.annotation.Configuration;
    import org.springframework.security.config.annotation.web.builders.HttpSecurity;
    import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
    import org.springframework.security.config.annotation.web.configuration.WebSecurityConfigurerAdapter;

    @Configuration
    @EnableWebSecurity
    public class SecurityConfig extends WebSecurityConfigurerAdapter {

      @Override
      protected void configure(HttpSecurity http) throws Exception {
        http
          .authorizeRequests()
            .antMatchers("/actuator/**").hasRole("ADMIN")
          .and()
          .httpBasic();
      }
    }

    6. How can you add custom information to the /actuator/info endpoint?

    Example Answer: You can add custom information to the /actuator/info endpoint by creating a @Configuration class that adds details to the InfoContributor bean:
    import org.springframework.boot.actuate.info.Info;
    import org.springframework.boot.actuate.info.InfoContributor;
    import org.springframework.stereotype.Component;

    @Component
    public class CustomInfoContributor implements InfoContributor {

      @Override
      public void contribute(Info.Builder builder) {
        builder.withDetail("customProperty", "customValue");
      }
    }
    This will add customProperty with customValue to the /actuator/info endpoint output.

    7. What is a health indicator in Spring Boot Actuator?

    Example Answer: A health indicator is a component that provides information about the health of a specific part of your application or its dependencies. It can be used to perform checks on databases, messaging systems, or other services your application depends on. You can use built-in health indicators or create custom ones by implementing the HealthIndicator interface.

    8. How do you create a custom health indicator?

    Example Answer: To create a custom health indicator, implement the HealthIndicator interface:
    import org.springframework.boot.actuate.health.Health;
    import org.springframework.boot.actuate.health.HealthIndicator;
    import org.springframework.stereotype.Component;

    @Component
    public class MyCustomHealthIndicator implements HealthIndicator {

      @Override
      public Health health() {
        // Custom health check logic
        boolean healthy = true; // replace with actual health check
        if (healthy) {
          return Health.up().withDetail("status", "up").build();
        } else {
          return Health.down().withDetail("status", "down").build();
        }
      }
    }
    This will add a custom health indicator to the /actuator/health endpoint.

    9. How do you view Actuator metrics in your application?

    Example Answer: Once Actuator is included and properly configured, you can view metrics by accessing the /actuator/metrics endpoint. This endpoint provides a comprehensive list of application metrics, including those related to JVM, application performance, and custom metrics.

    10. What is the purpose of the management.endpoints.web.exposure.include property?

    Example Answer: The management.endpoints.web.exposure.include property is used to specify which Actuator endpoints should be exposed over the web. By default, not all endpoints are exposed. This property allows you to configure which endpoints you want to be accessible via HTTP. For example:
    management.endpoints.web.exposure.include=health,info,metrics
    This configuration will expose the /actuator/health, /actuator/info, and /actuator/metrics endpoints.

    11. How do you view the list of available Actuator endpoints?

    Example Answer: To view the list of available Actuator endpoints, you can access the /actuator endpoint. This endpoint provides a map of all available endpoints and their respective paths. For example, you would navigate to http://localhost:8080/actuator in your browser or use a tool like curl:
    curl http://localhost:8080/actuator
    This will return a JSON object with all available endpoints and their paths.

    12. How do you change the default base path for Actuator endpoints?

    Example Answer: You can change the default base path for Actuator endpoints using the management.endpoints.web.base-path property in application.properties or application.yml. For example:

    In application.properties:

    management.endpoints.web.base-path=/admin

    In application.yml:

    management:
      endpoints:
        web:
          base-path: /admin
    With this configuration, the Actuator endpoints will now be accessible under /admin, e.g., /admin/health and /admin/metrics.

    13. How do you customize the response of the /actuator/health endpoint?

    Example Answer: You can customize the response of the /actuator/health endpoint by creating a custom health indicator or modifying the built-in ones. For example, to add custom details:

    1. Implement a custom health indicator:

    import org.springframework.boot.actuate.health.Health;
    import org.springframework.boot.actuate.health.HealthIndicator;
    import org.springframework.stereotype.Component;

    @Component
    public class CustomHealthIndicator implements HealthIndicator {

      @Override
      public Health health() {
        // Custom logic to check health
        return Health.up().withDetail("custom", "Everything is running smoothly").build();
      }
    }
    2. The custom health details will then be included in the /actuator/health response.

    14. What is the purpose of the management.endpoints.web.exposure.exclude property?

    Example Answer: The management.endpoints.web.exposure.exclude property allows you to exclude specific Actuator endpoints from being exposed over the web. For example:
    management.endpoints.web.exposure.exclude=metrics,env
    This configuration will exclude the /actuator/metrics and /actuator/env endpoints from being accessible via HTTP, even if they are defined in your application.

    15. How do you expose Actuator metrics in a format suitable for Prometheus?

    Example Answer: To expose Actuator metrics in a format suitable for Prometheus, you need to add the Micrometer Prometheus registry dependency and configure it:

    1. Add the Micrometer Prometheus dependency:

    <dependency>
      <groupId>io.micrometer</groupId>
      <artifactId>micrometer-registry-prometheus</artifactId>
    </dependency>

    2. Enable the Prometheus registry in application.properties:

    management.metrics.export.prometheus.enabled=true
    3. Metrics will be available at the /actuator/prometheus endpoint, which Prometheus can scrape.

    16. How do you use @Timed and @Counted annotations to monitor method execution?

    Example Answer: The @Timed and @Counted annotations from Micrometer allow you to monitor method execution and frequency.
    @Timed: Measures the time taken by a method to execute. Example:
    import io.micrometer.core.annotation.Timed;
    import org.springframework.stereotype.Service;

    @Service
    public class MyService {

      @Timed("myService.method.timer")
      public void myMethod() {
        // method implementation
      }
    }
    @Counted: Counts the number of times a method is called. Example:
    import io.micrometer.core.annotation.Counted;
    import org.springframework.stereotype.Service;

    @Service
    public class MyService {

      @Counted("myService.method.calls")
      public void anotherMethod() {
        // method implementation
      }
    }
    Metrics for these annotations will be available in the /actuator/metrics endpoint.

    17. How do you set up custom logging levels for Actuator endpoints?

    Example Answer: Custom logging levels for Actuator endpoints can be set using the logging.level property in application.properties or application.yml. For example, to set the logging level for health checks:

    In application.properties:

    logging.level.org.springframework.boot.actuate=DEBUG

    In application.yml:

    logging:
      level:
        org.springframework.boot.actuate: DEBUG
    This will enable DEBUG level logging for Actuator components, allowing you to see detailed logs related to Actuator operations.

    18. How can you view detailed information about the environment variables and configuration properties using Actuator?

    Example Answer: You can view detailed information about environment variables and configuration properties by accessing the /actuator/env endpoint. This endpoint provides details on environment properties, system properties, and configuration properties.
    To enable this endpoint, ensure it is included in your exposure settings:
    management.endpoints.web.exposure.include=env
    Accessing http://localhost:8080/actuator/env will display a JSON object with all the environment and configuration details.

    19. How do you create a custom Actuator endpoint?

    Example Answer: To create a custom Actuator endpoint, you can use the @Endpoint annotation:

    1. Define a new endpoint:

    import org.springframework.boot.actuate.endpoint.annotation.Endpoint;
    import org.springframework.boot.actuate.endpoint.annotation.ReadOperation;
    import org.springframework.stereotype.Component;

    @Component
    @Endpoint("custom")
    public class CustomEndpoint {

      @ReadOperation
      public String custom() {
        return "Custom endpoint response";
      }
    }
    2. Access this endpoint at /actuator/custom (if using default configuration).

    20. What is the difference between management.endpoints.web.exposure.include and management.endpoints.web.exposure.exclude?

    Example Answer:
    • management.endpoints.web.exposure.include: Specifies which Actuator endpoints should be included and exposed over HTTP. For example, management.endpoints.web.exposure.include=health,info will expose only the health and info endpoints.
    • management.endpoints.web.exposure.exclude: Specifies which Actuator endpoints should be excluded from being exposed. For example, management.endpoints.web.exposure.exclude=metrics,env will prevent these endpoints from being accessible even if they are defined.

    1. /actuator/health

    Purpose:
    The /actuator/health endpoint provides information about the health status of your application. This includes system checks, database connectivity, and any custom health indicators you have implemented.
    Default Behavior:
    Returns a simple JSON object indicating whether the application is "UP" or "DOWN". For example:
    { "status": "UP" }
    Configuration:
    • To include this endpoint in the response, ensure it is listed in the management.endpoints.web.exposure.include property:
    management.endpoints.web.exposure.include=health
    • To show detailed information, configure:
    management.endpoint.health.show-details=always

    2. /actuator/metrics

    Purpose:
    The /actuator/metrics endpoint exposes various metrics collected by the application, such as JVM metrics, HTTP requests, and custom metrics.
    Default Behavior:
    Returns a list of available metrics and their names. Example response:
    { "names": [ "jvm.memory.used", "http.server.requests", ... ] }
    Configuration:
    • To include this endpoint, add it to the exposure list:
    management.endpoints.web.exposure.include=metrics

    3. /actuator/env

    Purpose:
    The /actuator/env endpoint exposes environment properties and configuration properties of the application.
    Default Behavior:
    Returns a detailed view of environment properties, including system properties, environment variables, and configuration properties. Example:
    { "propertySources": [ ... ], "systemProperties": { "java.version": "11.0.9", ... }, "systemEnvironment": { "PATH": "/usr/local/bin", ... } }
    Configuration:
    • Ensure it is included in the exposure list:
    management.endpoints.web.exposure.include=env

    4. /actuator/info

    Purpose:
    The /actuator/info endpoint provides arbitrary application information, which can be customized with additional details such as build version, description, and custom metadata.
    Default Behavior:
    Returns a simple JSON object with application details. Example:

    Configuration:

    16. Configuration Summary

    To manage which endpoints are exposed and how they are configured, use properties in application.properties or application.yml. Here are some common properties for managing endpoints:
    # Include specific endpoints
    management.endpoints.web.exposure.include=health,info,metrics

    # Exclude specific endpoints
    management.endpoints.web.exposure.exclude=env,beans

    # Show detailed health information
    management.endpoint.health.show-details=always

    # Enable Prometheus metrics endpoint
    management.endpoints.web.exposure.include=prometheus

    17. /actuator/shutdown

    Purpose:
    The /actuator/shutdown endpoint allows you to gracefully shut down your Spring Boot application. This endpoint can be useful for integration with external systems or scripts that need to trigger application shutdown.
    Default Behavior:
    When accessed, this endpoint triggers a graceful shutdown of the application, allowing it to complete ongoing tasks before shutting down.
    Configuration:
    • To enable the shutdown endpoint, you need to include it in your application’s configuration:
    1. Add the dependency for actuator shutdown support:

    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
    2. Enable the shutdown endpoint: In application.properties or application.yml:
    management.endpoints.web.exposure.include=shutdown
    3. Optional - Secure the endpoint: It is highly recommended to secure this endpoint to prevent unauthorized access. For example, using Spring Security:
    import org.springframework.context.annotation.Bean;
    import org.springframework.context.annotation.Configuration;
    import org.springframework.security.config.annotation.web.builders.HttpSecurity;
    import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
    import org.springframework.security.config.annotation.web.configuration.WebSecurityConfigurerAdapter;

    @Configuration
    @EnableWebSecurity
    public SecurityConfig extends WebSecurityConfigurerAdapter {

        @Override
        protected void configure(HttpSecurity http) throws Exception {
            http
              .authorizeRequests()
                .antMatchers("/actuator/shutdown").hasRole("ADMIN")
              .and()
                .httpBasic();
        }
    }
    Example Usage: To shut down the application, send a POST request to /actuator/shutdown.

    18. /actuator/refresh

    Purpose:
    The /actuator/refresh endpoint is used to refresh the application’s configuration. This is particularly useful in scenarios where you use a configuration management tool like Spring Cloud Config and want to reload configuration changes without restarting the application.
    Default Behavior:
    When accessed, this endpoint triggers a refresh of the application context, reloading properties from the configuration source.
    Configuration:
    • To enable the refresh endpoint, you need to include it in your application’s configuration:
    1. Add the dependency for Spring Boot Actuator with Spring Cloud:

    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-config</artifactId>
    2. Enable the refresh endpoint: In application.properties or application.yml:
    management.endpoints.web.exposure.include=refresh
    3. Optional - Secure the endpoint: Similar to the shutdown endpoint, it is important to secure the refresh endpoint. For example:
    @Configuration
    @EnableWebSecurity
    public SecurityConfig extends WebSecurityConfigurerAdapter {

        @Override
        protected void configure(HttpSecurity http) throws Exception {
            http
              .authorizeRequests()
                .antMatchers("/actuator/refresh").hasRole("ADMIN")
              .and()
                .httpBasic();
        }
    }
    Example Usage: To refresh the application context, send a POST request to /actuator/refresh.
